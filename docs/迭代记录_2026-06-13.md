# Collection App 迭代记录

日期：2026-06-13

本文记录当前几轮产品和数据体系迭代，便于后续实现时对齐真实状态。

## 1. PWA 原型阶段

目标：先用 Web App / PWA 验证私人收藏管理的核心闭环。

已形成的方向：

1. 从原生 App 暂缓，转向 React + TypeScript + Vite PWA。
2. 建立仓库、Gallery、分析、设置四个主入口。
3. 支持藏品、购买记录、售出记录、分类、字段、标签、卡册等基础对象。
4. 支持本地优先和离线浏览，早期用 `localStorage`，后续升级到 IndexedDB。
5. CardHobby 抓取不直接绑死主 App，先作为独立 demo 输出 CSV。

## 2. Web 同步与图片阶段

目标：让网页端具备长期使用的数据稳定性基础。

已形成的方向：

1. 使用 IndexedDB 作为浏览器端主存储。
2. 使用 Supabase Auth 处理账号登录。
3. 使用 `collection_snapshots` 保存用户级数据快照。
4. 使用 Supabase Storage 上传图片原图，前端保留本地压缩预览。
5. 预留 `collection_operations` 操作日志，用于后续增量同步。
6. 设置页支持 JSON 备份导出。

当前判断：

1. 网络端仍需继续验证可靠性和冲突处理。
2. Obsidian vault 先承担可信本地数据仓角色。
3. Web Dataset 后续通过导入端口从 vault 获取结构化数据。

核心设计：

4. 订单通过 `items` 链接多个藏品。
5. 藏品可通过 `source_order` 辅助链接订单。
6. Obsidian 反链和 Dataview/Base 一起形成双链浏览。

## 4. Obsidian 模板与自动化阶段

目标：减少手动建档成本，同时保留人工确认。

已完成方向：

1. 建立分类模板：球星卡、拼豆、周边、其他手工、订单、图纸来源。
2. 所有分类包含 `tag details`，用于层级标签。
3. 球星卡字段包含卡牌公司、系列、限定编号、特卡名称、player、club、发售年份、赛季、卡价、是否带编、是否签字、是否物料。
4. 拼豆字段包含图纸、烫法。
5. 图片放入附件分类目录后，运行 Templater 脚本生成待入库 md。
6. 命名规则升级为四位连续编号，例如 `Card 0001`、`Perler 0001`、`Merch 0001`、`Handmade 0001`。
7. 录入完成后，用户手动把 md 从 `Pending` 拖到 `Items` 完成入库。

## 5. Obsidian Base、首页和统计阶段

目标：让 vault 不只是数据存储，也能成为漂亮的本地浏览入口。

已完成方向：

1. 建立藏品 Base 视图，用于直观展示全部藏品。
2. 首页从普通 Markdown 统计页升级为网页感 dashboard。
3. 首页标题迭代为 `MICEMMS`。
4. 加入 KPI、待入库、购买热力图、类别环、月度消费、平台消费、卖家统计、tag 花费等统计块。
5. 多轮优化宽屏、卡片尺寸、热力图、图表颜色、Properties 隐藏、订单 dashboard 和模板视觉。
6. 安装/等效实现 MCL 相关 CSS snippets，用于宽屏、多列和卡片视图。
