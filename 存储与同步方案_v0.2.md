# Collection App Storage And Sync Plan

## Current state

- `v0.1` persists collection data in the browser.
- Static assets are cached by the PWA service worker.
- User data currently lives in `localStorage`.

This is enough for demos, but it is not enough for long-term mobile use because:

1. `localStorage` capacity is small.
2. Image-heavy collections will grow fast.
3. Sync conflict handling does not exist yet.
4. There is no remote backup.

## Goal

Make the app behave like this:

1. Write locally first.
2. Read locally first.
3. When online, sync in the background.
4. When offline, keep browsing and editing.
5. When the device reconnects, push queued changes.

## Recommended architecture

### 1. Local database on phone

Use `IndexedDB` as the main local database.

- Better storage size than `localStorage`
- Works in installed PWA on iPhone and Android
- Suitable for structured data plus image metadata
- Can keep collection data cached on the phone

Suggested local stores:

- `categories`
- `fields`
- `tags`
- `items`
- `purchases`
- `sale_records`
- `digital_assets`
- `containers`
- `container_slots`
- `sync_queue`
- `sync_meta`

### 2. Cloud backend

Use `Supabase` first.

Why:

- Fast to ship
- Postgres data model fits this app
- Simple REST / JS client
- Row level security can support private mode
- Good fit for later image storage

Cloud tables should mirror the local data model:

- `categories`
- `fields`
- `tags`
- `items`
- `purchases`
- `sale_records`
- `digital_assets`
- `containers`
- `container_slots`

Common columns:

- `id`
- `user_id`
- `created_at`
- `updated_at`
- `deleted_at` for soft delete when needed
- `version` or `updated_at` for conflict resolution

### 3. Sync strategy

Use `local-first + outbox queue`.

Flow:

1. User edits data.
2. App writes to `IndexedDB` immediately.
3. App writes a sync job into `sync_queue`.
4. UI updates immediately from local data.
5. When online, the queue is processed.
6. Server response updates local records with latest sync metadata.

Recommended queue job shape:

```ts
type SyncJob = {
  id: string;
  entity: "item" | "purchase" | "tag" | "field" | "container" | "slot";
  entityId: string;
  action: "upsert" | "delete";
  payload: unknown;
  createdAt: string;
  retryCount: number;
};
```

### 4. Conflict policy

For private single-user mode, first use:

- `last write wins`
- compare `updated_at`

That is enough for `v0.2`.

Later, if multi-device editing becomes frequent, add:

- per-entity `version`
- merge prompts for rare hard conflicts

### 5. Image strategy

Do not keep full-resolution item photos in `localStorage`.

Recommended:

1. Compress on device before save
2. Keep preview and metadata locally
3. Upload originals or processed versions to cloud object storage

Use:

- local preview blob in `IndexedDB`
- remote file in Supabase Storage

## What can be cached on the phone?

Yes. The app can cache data on the phone.

In practice:

- PWA shell files: `Service Worker + Cache Storage`
- user records: `IndexedDB`
- image previews: `IndexedDB`

This means:

- the site can still open without network
- existing collection data can still be browsed
- locally edited records can wait for sync

## Development order

### Step 1

Replace current `localStorage` persistence with a persistence adapter:

- `loadSnapshot()`
- `saveSnapshot()`
- `enqueueSyncJob()`
- `flushSyncQueue()`

### Step 2

Switch adapter implementation to `IndexedDB`.

### Step 3

Add `sync_queue` and online listener.

### Step 4

Connect Supabase tables and push / pull sync.

### Step 5

Move images to object storage and keep compressed previews locally.

## Next coding task

The most practical next step is:

1. add a persistence adapter layer
2. migrate browser storage from `localStorage` to `IndexedDB`
3. add sync queue records for item and order changes

That gets us from "demo persistence" to "real mobile-first offline storage".
