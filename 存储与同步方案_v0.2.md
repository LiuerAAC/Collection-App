# Collection App Storage And Sync Plan

Updated: 2026-06-13

## Current state

- The PWA persists collection data in the browser with IndexedDB.
- Static assets are cached by the PWA service worker.
- Legacy `localStorage` is only a migration source for old guest data.
- Supabase Auth, snapshot sync, and Storage upload are now documented and partially implemented.
- The user's most trusted current dataset lives first in an Obsidian vault.

The Obsidian vault path is:

```text
/Users/yuxuancao/Library/Mobile Documents/iCloud~md~obsidian/Documents/Life
```

Browser local storage plus snapshot sync is enough for app usage, but not enough as the only source of truth yet because:

1. Network sync stability still needs more real-world validation.
2. Image-heavy collections will grow fast.
3. Sync conflict handling is still simple.
4. The vault already contains real structured data, templates, orders, and image workflows.
5. The Web Dataset needs a controlled way to import from the vault.

## Goal

Make the app behave like this:

1. Write locally first.
2. Read locally first.
3. When online, sync in the background.
4. When offline, keep browsing and editing.
5. When the device reconnects, push queued changes.
6. When Obsidian has the latest trusted records, scan and import them into the Web Dataset through a previewable port.

## Recommended architecture

### 1. Local database on phone

Use `IndexedDB` as the main local database.

- Better storage size than `localStorage`
- Works in installed PWA on iPhone and Android
- Suitable for structured data plus image metadata
- Can keep collection data cached on the phone

Suggested local stores:

- `categories`
- `fields`
- `tags`
- `items`
- `purchases`
- `sale_records`
- `digital_assets`
- `containers`
- `container_slots`
- `sync_queue`
- `sync_meta`
- `obsidian_import_batches`
- `obsidian_source_refs`
- `bead_colors`
- `bead_inventory_logs`

### 2. Cloud backend

Use `Supabase` first.

Why:

- Fast to ship
- Postgres data model fits this app
- Simple REST / JS client
- Row level security can support private mode
- Good fit for later image storage

Cloud tables should mirror the local data model:

- `categories`
- `fields`
- `tags`
- `items`
- `purchases`
- `sale_records`
- `digital_assets`
- `containers`
- `container_slots`
- `obsidian_import_batches`
- `obsidian_source_refs`
- `bead_colors`
- `bead_inventory_logs`

Common columns:

- `id`
- `user_id`
- `created_at`
- `updated_at`
- `deleted_at` for soft delete when needed
- `version` or `updated_at` for conflict resolution

### 3. Sync strategy

Use `local-first + outbox queue`.

Flow:

1. User edits data.
2. App writes to `IndexedDB` immediately.
3. App writes a sync job into `sync_queue`.
4. UI updates immediately from local data.
5. When online, the queue is processed.
6. Server response updates local records with latest sync metadata.

Recommended queue job shape:

```ts
type SyncJob = {
  id: string;
  entity: "item" | "purchase" | "tag" | "field" | "container" | "slot";
  entityId: string;
  action: "upsert" | "delete";
  payload: unknown;
  createdAt: string;
  retryCount: number;
};
```

### 4. Obsidian vault import port

The vault is a local-first upstream source, not a cloud backend. The app should treat it as an import source with explicit user confirmation.

Recommended flow:

1. User selects or confirms the vault path.
2. App scans known directories:
   - `10_Collections/Items/**`
   - `10_Collections/Pending/**`
   - `10_Collections/Orders/**`
   - `10_Collections/Sources/**`
   - `20_Attachments/**`
3. App parses Markdown frontmatter / Properties.
4. App parses Obsidian links and image references.
5. App maps records into Dataset entities.
6. App compares against existing `obsidian_source_refs`.
7. App shows an import preview:
   - new
   - updated
   - skipped
   - conflicts
   - failed
8. User confirms upload.
9. App writes Dataset records, uploads assets, and records an import batch.

The first implementation should not delete remote records. Deletion sync can be added later with explicit review.

Recommended source reference:

```ts
type ObsidianSourceRef = {
  id: string;
  entityType: "item" | "purchase" | "source" | "asset";
  entityId: string;
  vaultPath: string;
  fileName: string;
  fileMtime?: string;
  contentHash?: string;
  importBatchId: string;
  lastImportedAt: string;
};
```

Matching order:

1. Existing `ObsidianSourceRef.vaultPath`
2. Explicit external id if later added to frontmatter
3. `Purchase.orderNo`
4. File hash for assets
5. Manual conflict resolution

### 5. Dataset bead inventory

MARD bead inventory belongs in the structured Dataset, not only in Obsidian text notes.

Recommended local/cloud stores:

1. `bead_colors`
2. `bead_inventory_logs`
3. `pattern_usage_results`

`bead_colors` should contain the 221 MARD colors and group filters:

```text
All, A, B, C, D, E, F, G, H, M
```

Minimal shape:

```ts
type BeadColor = {
  id: string;
  brand: "MARD";
  code: string;
  groupCode: "A" | "B" | "C" | "D" | "E" | "F" | "G" | "H" | "M";
  name?: string;
  hex?: string;
  rgb?: [number, number, number];
  swatchUrl?: string;
  stockCount: number;
  totalAdded: number;
  totalUsed: number;
  updatedAt: string;
};
```

Inventory changes must be append-only logs:

```ts
type BeadInventoryLog = {
  id: string;
  beadColorId: string;
  action: "add" | "consume" | "adjust";
  quantity: number;
  itemId?: string;
  sourcePatternId?: string;
  source: "manual" | "pattern_recognition" | "import" | "correction";
  notes?: string;
  createdAt: string;
};
```

Adding stock from the library UI:

1. User clicks the plus button on a bead color.
2. App adds `1000` to `stockCount`.
3. App writes a `BeadInventoryLog` with action `add`.

Consuming stock from pattern recognition:

1. Pattern recognition creates a draft usage result.
2. User reviews and corrects color counts.
3. App writes `consume` logs.
4. App subtracts quantities from `stockCount`.
5. App links logs to the perler item and source pattern.

### 6. Conflict policy

For private single-user mode, first use:

- `last write wins`
- compare `updated_at`

That is enough for `v0.2`.

Later, if multi-device editing becomes frequent, add:

- per-entity `version`
- merge prompts for rare hard conflicts

### 7. Image strategy

Do not keep full-resolution item photos in `localStorage`.

Recommended:

1. Compress on device before save
2. Keep preview and metadata locally
3. Upload originals or processed versions to cloud object storage

Use:

- local preview blob in `IndexedDB`
- remote file in Supabase Storage

## What can be cached on the phone?

Yes. The app can cache data on the phone.

In practice:

- PWA shell files: `Service Worker + Cache Storage`
- user records: `IndexedDB`
- image previews: `IndexedDB`

This means:

- the site can still open without network
- existing collection data can still be browsed
- locally edited records can wait for sync
