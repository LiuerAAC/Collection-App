# Collection App 开发文档

日期：2026-06-13
形态：Web App / PWA
目标：用 PWA 完成私人收藏管理核心流程，支持本地优先、离线浏览、Supabase 同步，并在下一阶段接入 Obsidian vault 数据端口

## 1. 当前技术架构

1. 前端：React + TypeScript + Vite
2. PWA：`manifest.webmanifest` + `sw.js`
3. 本地存储：IndexedDB
4. 状态管理：React Context
5. 云同步：Supabase Auth + `collection_snapshots`
6. 图片存储：IndexedDB 暂存原图，Supabase Storage 上传远端图片
7. 构建输出：`dist/`

当前本地持久化入口：

1. `src/storage/persistence.ts`
2. `src/storage/indexedDb.ts`
3. `src/storage/localAssets.ts`
4. `src/storage/cloudSync.ts`

下一阶段新增：

1. Obsidian vault 读取端口。
2. Markdown / YAML / Obsidian 内链解析。
3. Dataset 导入预览和上传。
4. MARD 221 色拼豆库存数据模型。

## 2. 当前页面结构

### 2.1 顶部导航

1. `Collection App`
2. 仓库
3. Gallery
4. 分析
5. 设置

说明：

1. 顶部导航固定在页面上方。
2. 四个主入口使用圆形图标。
3. 桌面和手机端共用同一套导航结构。

### 2.2 仓库

仓库页已拆成三个子页面：

1. 默认仓库
2. 新增藏品
3. 订单管理

#### 默认仓库

1. 默认显示全部藏品。
2. 支持按分类筛选。
3. 支持按标签筛选。
4. 每个藏品卡片默认只显示图片。
5. 点击藏品进入详情弹层。

#### 藏品详情

1. 展示图片。
2. 展示状态、卡价、购买时间、购买金额、模板字段等信息。
3. 支持修改。
4. 支持删除。
5. 修改图片支持从图库选择或拍照上传。

#### 新增藏品

1. 先选择类别。
2. 默认重点支持卡牌流程。
3. 图片与文本信息改为上下排列。
4. 上传图片统一缩放到固定展示尺寸。
5. 卡牌类字段包括：
   - 卡牌公司
   - 系列
   - 年份
   - 球员名称
   - 队伍
   - 卡牌编号
   - 折射类型
   - 特卡名称
   - 是否带编
   - 编号
   - 是否物料
   - 是否签字
   - 是否大比例
   - 卡价
6. 卡牌公司是下拉选项。
7. 系列、球员名称、队伍支持基于已有数据的自动补全。

#### 订单管理

1. 保留订单列表布局。
2. 点击订单后弹出订单详情页。
3. 订单支持增删改查。
4. 支持将一个订单绑定到多个藏品。
5. 当绑定成功后，如果藏品的购买时间或购买金额为空，会自动同步订单中的对应信息。
6. 新增订单功能已具备基础版本。

### 2.3 Gallery

1. 支持卡册展示。
2. 支持选择卡册和页码。
3. 支持将藏品放入固定卡位。
4. 一个藏品可以出现在多个展示场景中。

### 2.4 分析

当前已支持：

1. 藏品总数
2. 商品金额
3. 邮费
4. 总金额
5. 售出总额
6. 标签数量
7. 按大类统计

待新增：

### 2.5 设置

当前已支持：

1. 分类模板编辑
2. 字段管理
3. 全局标签编辑

交互特征：

1. 分类管理、字段管理、全局标签均为可折叠模块。
2. 字段默认折叠展示，只显示字段名称和类型。
3. 点击字段后展开编辑内容。
4. 字段支持拖动调整顺序。
5. 模板变更会立即影响仓库新增藏品表单。

## 4. 本地存储与离线能力

1. 藏品、订单、模板、标签、卡册等数据写入 IndexedDB。
2. 页面资源通过 `Service Worker` 做基础缓存
3. 在 `localhost` 开发环境中自动关闭旧的 `Service Worker` 注册，避免缓存旧页面
4. 旧版 `localStorage` 数据仅作为 `guest` scope 的迁移来源。
5. 图片原图先写入 IndexedDB 的 pending asset 记录，上传成功后替换为远端 URL。
6. Sync queue 记录待推送动作和失败重试状态。

## 4.1 Supabase 同步能力

当前同步设计：

1. 账号使用 Supabase Auth 邮箱密码登录。
2. 前端将 `datasetId` 绑定为当前 `user.id`。
3. 数据快照写入 Supabase `collection_snapshots`。
4. 图片上传到 `collection-assets` bucket。
5. 配置从环境变量读取，不在 UI 中暴露项目级密钥配置。
6. 可选增量表 `collection_operations` 已写入配置文档，后续用于操作日志同步。

已知限制：

1. 当前以快照同步为主，不是完整增量同步。
2. 冲突处理仍偏简单。
3. 图片 bucket 当前按 public URL 方案设计。
4. 还没有从 Obsidian vault 读取数据的实现。
