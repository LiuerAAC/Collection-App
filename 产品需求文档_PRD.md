# 收藏管理程序 PRD

版本：v0.4
日期：2026-06-13
定位：个人收藏管理、数字化展示和收藏数据分析工具
平台：MVP 先做 Web App / PWA，移动端原生 App 暂缓

## 1. 产品背景

用户的收藏对象不只限于球星卡，还包括自制手工制品、周边、徽章、拼豆、钩织等多类藏品。当前需求不是单纯的“相册”或“购物记录本”，而是希望把“购买记录、藏品信息、图片、分类标签、展示位置、统计分析”连接成一个可长期维护的个人收藏系统。

产品需要支持本地使用和云端同步，让用户可以通过 PWA 在浏览器里完成录入、查看和整理；有网络时可同步或拉取更新，无网络时也能继续浏览本地已缓存内容。

当前产品先按私人收藏工具设计，不在 MVP 阶段提供公开社区、公开分享或多人协作能力。

当前开发策略从原生 App 转向 PWA，核心原因是希望先避开 iOS 开发者账号、打包签名和审核流程成本，把精力集中在产品结构、数据模型和实际使用体验上。

2026-06-13 更新：由于网络端同步和远端存储仍需要继续验证稳定性，当前真实数据先在本地 Obsidian vault 中维护。Web App 下一阶段需要新增 Obsidian 数据端口，从 vault 读取藏品、订单、图纸来源、图片和待入库数据，解析匹配后上传到网页 Dataset。Dataset 同时新增 MARD 221 色拼豆库存，用于拼豆图纸识别后的库存扣减和色号使用排行。

## 2. 产品目标

### 2.1 核心目标

1. 帮用户建立一个可持续维护的个人藏品数据库。
2. 支持多类型藏品，不把系统锁死在“卡牌”一种模型里。
3. 将购买记录和藏品本体信息分离管理，并允许二者关联。
4. 提供数字化 gallery，让收藏不只是存档，也能被整理和展示。
5. 提供基础数据分析，帮助用户了解收藏数量、金额、类别分布和标签分布。

### 2.2 非目标

以下内容不作为第一阶段核心目标：

1. 公开社区、公开分享、交易、估价市场。
2. 多用户协作收藏。
3. 复杂 3D 渲染展示。
4. 完全自动化识别所有藏品类型。
5. 对外部购物网站做深度自动抓取和长期稳定维护。

## 3. 目标用户

### 3.1 核心用户

有长期收藏习惯的个人用户，尤其是：

1. 球星卡收藏者。
2. 手工制品收藏或创作者。
3. 徽章、周边、拼豆、钩织等多品类收藏者。
4. 希望对收藏做归档、展示和数据统计的人。

### 3.2 用户特征

1. 收藏品类可能很多，且每类字段不同。
2. 希望录入过程尽量轻量，不想每次都填写大量固定字段。
3. 对展示有审美要求，希望 gallery 更像“整理自己的收藏空间”，而不是普通列表。
4. 可能已经有历史购买记录，需要未来支持批量导入。

## 4. 产品原则

1. PWA 优先：先让浏览器版本成为真实可用的日常工具，再决定是否做原生封装。
2. 数据模型灵活：藏品大类和字段可以自定义，避免后期被固定结构卡住。
3. 先闭环后炫技：MVP 先完成录入、查看、筛选、展示和同步，再做高级识别、3D 展示和 AI 分析。
4. 手动优先，自动增强：所有自动识别能力都必须有手动兜底。
5. gallery 是产品气质核心：它不是附属相册，而是收藏管理的第二入口。

## 4.1 已确认产品决策

1. MVP 先做 Web App / PWA。
2. MVP 按私人收藏工具设计。
3. 一个购买记录可以对应多个藏品。
4. 一个藏品暂时只支持一条购买记录和一条售出记录。
5. Gallery 中一个藏品可以被放入多个展示场景。
6. 金额统计包含邮费，同时分别显示商品金额、邮费和总金额。
7. 需要记录藏品当前实体存放位置。
8. 需要提供默认字段模板，第一重点是卡牌模板。
9. CardHobby 数据导入先独立制作抓取 demo，并将抓取结果导出为 CSV，作为后续导入和数据测试基础。
10. 离线浏览是 v0.1 的硬需求，在线同步能力先做架构预留和状态设计。
11. Obsidian vault 暂时作为可信本地数据源，路径为 `/Users/yuxuancao/Library/Mobile Documents/iCloud~md~obsidian/Documents/Life`。
12. Web App 需要支持从 Obsidian vault 导入/上传结构化数据，而不是要求用户立即放弃本地 vault。
13. Dataset 将新增 MARD 221 色拼豆库，支持库存增减、图纸消耗扣减和色号使用排行。

## 5. 信息架构

产品分为三个主模块：

1. 仓库：管理所有藏品和购买记录。
2. Gallery：数字化展示藏品。
3. 数据分析：按类别、标签、价格、数量等维度统计。

v0.4 起，补充两个数据模块：

1. Obsidian 数据端口：从 vault 读取 Markdown、Properties、内链、附件和 Base 辅助配置，映射到 Web Dataset。
2. Dataset 拼豆库：维护 MARD 色号、库存、消耗日志和图纸关联。

PWA 推荐主导航：

1. 仓库
2. Gallery
3. 分析
4. 设置

PWA 中先直接承载日常使用和管理，不再把网页端只当后台：

1. 藏品表格
2. 购买记录表格
3. Gallery 编辑器
4. 字段与分类配置
5. 数据分析

## 6. 核心对象定义

### 6.1 藏品 Item

藏品是用户实际收藏的对象，例如一张卡、一枚徽章、一个拼豆作品。

基础字段：

1. 名称
2. 大类
3. 小类
4. 图片
5. 标签
6. 状态
7. 描述/备注
8. 当前实体存放位置
9. 关联购买记录
10. 关联售出记录
11. 创建时间
12. 更新时间

推荐状态：

1. 已拥有
2. 在途
3. 已售出
4. 已交换

### 6.2 购买记录 Purchase

购买记录记录一次购买行为，可以关联一个或多个藏品。对于单个藏品，MVP 暂时只允许关联一条购买记录。

基础字段：

1. 商家
2. 购买平台
3. 付款日期
4. 商品标题或订单标题
5. 商品金额
6. 邮费
7. 总金额
8. 订单截图或链接
9. 备注
10. 关联藏品

### 6.2.1 售出记录 Sale Record

售出记录记录某个藏品离开收藏系统的交易结果。MVP 中一个藏品最多关联一条售出记录。

基础字段：

1. 售出日期
2. 售出平台
3. 买家/对象
4. 售出金额
5. 邮费
6. 总金额
7. 截图或链接
8. 备注

### 6.3 分类 Category

大类用于区分藏品类型，例如：

1. 卡牌
2. 徽章
3. 拼豆
4. 钩织
5. 周边
6. 手工制品

大类必须支持用户自定义,可以包含几类预设模版。

### 6.4 自定义字段 Custom Field

不同大类下有不同字段。字段应由用户配置，而不是写死在系统里。

字段类型建议：

1. 文本
2. 数字
3. 金额
4. 日期
5. 单选
6. 多选
7. 开关
8. 图片
9. 链接

卡牌类推荐字段：

1. 卡牌公司
2. 系列
3. 球员名称
4. 球队
5. 年份
6. 卡牌编号
7. 折射类型
8. 限定编号
9. 是否特卡
10. 特卡系列
11. 是否签字
12. 是否物料
13. 品相/评级

拼豆类推荐字段：

1. 作品主题
2. 烫法
3. 尺寸
4. 图纸来源

徽章类推荐字段：

1. 主题
2. 工艺
3. 材质
4. 尺寸

### 6.5 标签 Tag

Tag 是跨大类共享的信息，用于解决“卡和徽章都可能属于足球”这类问题。

示例：

1. 足球
2. 曼城
3. 哈兰德

Tag 不属于某个大类，而是全局共享。一个藏品可以有多个 tag。

### 6.6 Gallery

Gallery 是数字化展示空间，也是产品的情绪价值和交互体验核心。它不只是“看图列表”，而是让用户像整理实体收藏一样，摆放、翻看和管理自己的数字藏品。

MVP 中先支持“卡册”一种展示容器，卡册位置相对固定，适合卡牌类藏品。后续扩展展示架、毛毡板、墙面、桌面等自由展示场景，并逐步支持拖拽、旋转、自动吸附和更游戏化的整理体验。

基础对象：

1. Gallery 场景
2. 展示容器
3. 页面
4. 位置
5. 被放入的藏品
6. 展示参数

同一个藏品可以出现在多个 Gallery 场景中，因为数字展示不等同于实体占位。

### 6.7 数字化资产 Digital Asset

数字化资产是从藏品图片进一步处理得到的展示基础。它和普通图片不同，不只是“上传一张图”，而是尽量把藏品变成可用于 Gallery 的干净展示素材。

MVP 建议先实现轻量版本：

1. 正面图片
2. 背面图片
3. 自动或手动裁切边界
4. 透视校正
5. 背景弱化或移除
6. 标准展示比例
7. 原图保留

后续可以进一步迭代：

1. 卡牌边缘厚度模拟
2. 反光和折射效果模拟
3. 2.5D 展示素材
4. 非标准尺寸记录
5. 用于展板、展示架的可旋转展示资产

这部分是 Gallery 质感的前置基础，但不应在 MVP 阶段追求完整 3D 重建。

### 6.8 Obsidian Vault 数据源

Obsidian vault 是当前真实数据的本地先行存储。Web App 不直接替代它，而是通过数据端口读取并上传。

Vault 路径：

```text
/Users/yuxuancao/Library/Mobile Documents/iCloud~md~obsidian/Documents/Life
```

优先读取范围：

1. `10_Collections/Items/**`：已入库藏品。
2. `10_Collections/Pending/**`：待入库藏品草稿。
3. `10_Collections/Orders/**`：订单。
4. `10_Collections/Sources/**`：图纸来源。
5. `20_Attachments/**`：图片、图纸和附件。
6. `.base` 文件：用于辅助理解本地视图，不作为唯一数据来源。

解析内容：

1. YAML frontmatter / Obsidian Properties。
2. Obsidian 内链。
3. 图片引用。
4. 文件路径所属分类。
5. 文件名、更新时间和附件路径。

匹配原则：

1. `vaultPath` 是第一版最稳定的本地唯一线索。
2. 订单优先按 `order_no` 匹配，没有订单号时按 `vaultPath` 匹配。
3. 藏品优先按 `vaultPath` 匹配，后续可引入显式 `externalId`。
4. 图片按相对路径和文件 hash 匹配。
5. 重复导入应更新已有记录，不生成副本。

### 6.9 MARD 拼豆色号

MARD 拼豆色号是拼豆库存管理的基础对象。第一版目标是建立 221 色完整色库，并支持按色号组筛选、补库存、扣库存和统计排行。

基础字段：

1. 色号。
2. 色号组。
3. 颜色名称。
4. HEX 色值。
5. RGB 色值。
6. 色卡图片或纯色色块。
7. 当前库存。
8. 累计入库。
9. 累计消耗。
10. 最近更新时间。

筛选组：

1. 全部
2. A
3. B
4. C
5. D
6. E
7. F
8. G
9. H
10. M

### 6.10 拼豆库存日志

拼豆库存变化必须形成日志，避免库存数字无法追溯。

日志字段：

1. id
2. 色号
3. 操作类型：入库、消耗、盘点校正
4. 数量
5. 关联拼豆藏品
6. 关联图纸
7. 来源：手动加库存、图纸识别、人工修正
8. 创建时间
9. 备注

## 7. 用户流程

### 7.1 新增藏品

1. 用户点击新增。
2. 选择藏品大类。
3. 填写基础信息。
4. 根据大类填写自定义字段。
5. 上传或拍摄图片。
6. 对图片进行数字化处理，包括裁切、透视校正、正反面记录和展示比例确认。
7. 填写当前实体存放位置。
8. 选择是否关联购买记录。
9. 保存后进入仓库列表。

### 7.2 新增或导入购买记录并关联藏品

1. 用户点击新增购买记录。
2. 填写商家、日期、金额、邮费、平台等信息。
3. 可上传订单截图或粘贴订单链接。
4. 可选择已有藏品进行关联。
5. 也可以在购买记录下创建新藏品。

批量导入流程：

1. 用户先通过独立 CardHobby 抓取 demo 获取订单数据。
2. demo 将抓取结果导出为 CSV。
3. App 或后续网页端读取 CSV。
4. 用户确认商家、付款日期、金额、邮费、商品标题等字段。
5. 系统按订单生成购买记录。
6. 用户将购买记录批量或逐条关联到藏品。
7. 无法识别或缺失字段的记录进入待处理列表。

### 7.3 维护分类和字段

1. 用户进入设置。
2. 新增或编辑大类。
3. 为大类配置字段。
4. 为单选/多选字段配置选项。
5. 保存后，新增藏品时自动出现对应字段。

### 7.4 将卡牌放入卡册

1. 用户进入 Gallery。
2. 新建卡册。
3. 配置卡册名称、外壳颜色、页面样式、每页卡位数量、是否支持正反面。
4. 进入卡册页面。
5. 点击空位。
6. 从仓库选择藏品。
7. 藏品被放入对应位置。
8. 用户可以移动、移除或替换。

### 7.5 查看数据分析

1. 用户进入分析页。
2. 查看总藏品数、总购买金额、品类分布。
3. 按大类、tag、小类筛选。
4. 查看数量和金额统计。

#### 8.1.8 CardHobby 购买记录导入

优先级：P1

说明：

订单数量较多时，粘贴文本的工作量过大，因此需要优先验证 CardHobby 抓取方案。但从 CardHobby 等网站抓取订单信息存在稳定性、登录态、反爬、隐私和维护成本问题，不建议直接作为 App MVP 的内置强依赖。

第一阶段方案：

1. 独立制作 CardHobby 抓取 demo。
2. demo 用于验证登录、翻页、订单字段抓取、稳定性和数据完整性。
3. 抓取结果先导出 CSV。
4. CSV 作为 App 导入能力和后续测试的数据基础。
5. App 内先实现 CSV 导入和字段确认。
6. 等 demo 稳定后，再决定是否内置 CardHobby 登录抓取。

CSV 推荐字段：

1. 平台
2. 商家

#### 8.1.9 藏品数字化扫描

优先级：P1

说明：

数字化扫描是 Gallery 后续展示效果的基础。MVP 不追求完整 3D 重建，但新增藏品时应尽量把图片处理成可展示资产。

功能：

1. 拍摄或上传正面图。
2. 拍摄或上传背面图。
3. 自动识别边界。
4. 支持手动调整裁切边界。
5. 进行透视校正。
6. 保留原图和处理后图片。
7. 记录标准展示比例和非标准尺寸备注。

#### 8.1.10 卡牌拍摄识别

优先级：P2

说明：

卡牌正反面识别是高价值功能，但识别准确率和模型成本不确定，不建议放入 MVP。

后续能力：

1. 拍摄正面和背面。
2. 自动裁切卡片区域。
3. 识别球员名、俱乐部、系列、卡号等信息。
4. 识别后进入确认页。
5. 对非常规尺寸卡片支持手动调整尺寸或裁切边界。

#### 8.1.11 Obsidian 数据端口

优先级：P0

说明：

当前真实数据先存储在 Obsidian vault。Web App 需要提供一个可控、可预览、可重复运行的数据端口，将 vault 中的结构化数据上传到 Dataset。

功能：

1. 读取指定 vault 路径。
2. 扫描藏品、待入库、订单、图纸来源和附件目录。
3. 解析 Markdown frontmatter、Obsidian Properties、内链和图片引用。
4. 将 vault 分类映射到 Web App 分类和 Dataset。
5. 识别新增、更新、跳过和冲突数据。
6. 上传前展示导入预览。
7. 用户确认后上传结构化数据和图片资产。
8. 写入导入日志。
9. 支持重复运行，不重复创建同一条记录。
10. 第一版不自动删除 Web 端数据。

验收标准：

1. 能从 `Life` vault 读取订单和藏品。
2. 能保留订单与藏品的双向关联关系。
3. 能把图纸来源关联到拼豆或手工藏品。
4. 能输出失败项列表，不因单个文件失败中断整批导入。

## 13. 体验要求

### 13.1 录入体验

1. 新增藏品时先选择大类，再出现对应字段。
2. 常用字段和选项应尽量可复用。
3. 新建 tag 时不应打断当前录入流程。
4. 保存后可以继续添加下一个藏品。
5. 拍摄后进入轻量数字化处理流程，但允许用户跳过。
6. 自动裁切或识别结果必须可编辑。
7. 实体存放位置应是可选字段，但推荐在卡牌模板中默认出现。

### 13.2 Gallery 体验

1. 卡册需要有明确的翻页感。
2. 空位、已放置、选中、拖动状态需要清晰。
3. 用户可以从仓库中挑选未放入卡册的藏品。
4. 一个藏品允许放进多个 Gallery 场景。
5. 卡册第一版以固定卡位为主。
6. 展板等后续容器以拖拽、旋转和自由摆放为主。
7. Gallery 优先展示数字化处理后的素材，原图作为兜底。

### 13.3 分析体验

1. 先给简单结果，不要求用户配置复杂查询。
2. 筛选条件和统计结果需要联动。
3. 金额统计包含邮费。
4. 需要同时显示商品金额、邮费和总金额。
5. 购买成本和售出金额分开展示。

## 14. 已确认问题和待继续确认问题

### 14.1 已确认

1. MVP 先做 Web App / PWA。
2. MVP 按私人收藏工具设计。
3. 一个购买记录支持对应多个藏品。
4. 一个藏品暂时只支持一条购买记录和一条售出记录。
5. Gallery 中一个藏品可以被放入多个展示场景。
6. 金额统计包含邮费，并同时显示商品金额、邮费和总金额。
7. CardHobby 先独立制作抓取 demo，抓取结果导出 CSV，作为后续测试和导入基础。
8. 需要记录藏品当前实体存放位置。
9. 需要提供默认字段模板，第一重点是卡牌模板。
10. Gallery 后续方向是类似交互游戏的展示体验；卡册偏固定卡位，展板等容器后续支持拖拽、旋转和自由摆放。
11. 当前阶段优先避免原生 iOS 成本和审核流程，先用 PWA 承担真实使用场景。
12. 有网络时更新数据，没有网络时浏览本地内容，是产品当前的核心使用场景。
13. 当前真实数据先存放在 Obsidian vault，Web App 需要通过数据端口接入。
14. MARD 221 色拼豆库是 Dataset 下一阶段的重点功能。

### 14.2 仍需确认

1. 售出记录是否需要参与利润统计。建议：v1.0 再做利润，MVP 只记录售出金额和状态。
2. 实体存放位置是否需要结构化。建议：MVP 先文本字段，后续再做“柜子/盒子/卡册/页码/位置”的层级模型。
3. 数字化扫描的第一版是否要求自动背景移除。建议：MVP 可先做裁切和透视校正，背景移除作为可选增强。
4. CardHobby demo 是否需要处理登录态和验证码。建议：先用可控测试账号和手动登录，验证抓取结果质量。
5. v0.2 的远端同步是“自动同步”还是“手动点击同步”。建议：先手动同步，便于控制冲突和调试。
6. MARD 221 色的权威色值来源需要确认。建议：第一版允许手动维护或导入色卡表，避免依赖不稳定网页抓取。
7. 图纸识别准确率和扣库存是否允许自动确认。建议：第一版必须人工确认。

## 15. 开发建议

### 15.1 技术方向

当前阶段推荐技术方向：

1. 前端：React + TypeScript + Vite。
2. PWA：Manifest + Service Worker。
3. 本地存储：IndexedDB 为主，localStorage 仅作为旧数据迁移来源或轻量配置。
4. 后端和同步：优先 Supabase Auth、Postgres snapshot、Storage。
5. 图片存储：本地预览 + 远端对象存储。
6. Obsidian 导入：第一版以本地文件扫描、Markdown 解析和差异预览为主。

如果目标是最快 vibe coding 出 v0.1，推荐：

1. 先把仓库、Gallery、分析、设置四个主视图跑通。
2. 先做本地优先和离线浏览，再补在线同步。
3. 使用现成后端服务减少账号和同步成本。
4. CardHobby 抓取 demo 独立于主 App 开发，避免外部网站不稳定影响主线进度。

### 15.2 实现顺序

1. 建立数据模型。
2. 实现分类和字段配置。
3. 实现藏品 CRUD。
4. 实现购买记录 CRUD。
5. 实现售出记录 CRUD。
6. 实现藏品、购买记录、售出记录关联。
7. 实现搜索、筛选和 tag。
8. 实现基础数字化扫描处理。
9. 实现基础 Gallery 卡册。
10. 实现数据分析。
11. 实现同步。
12. 独立实现 CardHobby 抓取 demo。
13. 实现 CardHobby CSV 导入确认流程。
14. 实现 Obsidian vault 数据端口。
15. 实现 MARD 221 色拼豆库。
16. 实现拼豆图纸识别后的库存扣减确认流程。

### 15.3 第一批原型验收标准

1. 可以新增一个“卡牌”大类并配置字段。
2. 可以新增一张卡牌，上传图片，并填写卡牌公司、系列、球员、编号等字段。
3. 可以对卡牌图片进行基础裁切和透视校正，并保留正反面图。
4. 可以记录这张卡牌的实体存放位置。
5. 可以新增一条购买记录，并关联这张卡牌。
6. 可以新增一条售出记录，并关联这张卡牌。
7. 可以在仓库中通过 tag 或关键词找到这张卡牌。
8. 可以新建一个卡册，并把这张卡牌放进指定位置。
9. 可以在另一个 Gallery 场景或卡册中再次放入同一张卡牌。
10. 可以在分析页看到卡牌数量、商品金额、邮费和总金额变化。
11. 可以运行 CardHobby demo 生成 CSV，并将 CSV 导入为购买记录测试数据。

## 16. 风险和注意事项

1. 自定义字段过度复杂会拖慢 MVP，需要先限制字段类型。
2. 自动抓取外部网站维护成本高，需要先用独立 demo 验证，不应直接绑死主 App。
3. 图片和同步会明显增加存储成本，需要尽早确定图片压缩策略。
4. 数字化扫描如果一开始追求高精度重建，会影响核心管理功能上线。
5. Gallery 如果一开始追求复杂渲染、自由拖拽和旋转，会影响核心管理功能上线。
6. AI 分析必须依赖足够数据量，否则容易变成演示功能。

## 17. 当前推荐结论

MVP 应围绕“私人收藏数据库 + 购买/售出记录关联 + 本地优先 + 离线浏览 + 简单卡册展示 + 基础统计 + CardHobby CSV 测试导入 + 在线同步预留”来做。v0.4 则应围绕“Obsidian 本地数据端口 + Web Dataset 承接 + MARD 拼豆库存”来做。

最重要的产品判断是：这款产品的底层不是“卡牌 App”，而是“可自定义字段的多品类收藏管理系统”。当前最合适的落地方式不是原生 App，而是可安装、可离线浏览、可逐步补齐同步能力的 PWA。卡牌是第一类重点场景，Gallery 是让产品有情绪价值的展示层，数据分析是让长期维护变得有回报的反馈层。
